# GitHub Issue Factory

The issue factory packages the Sidekick-owned Codex loop as two release assets:

- `.github/actions/issue-factory-run/action.yml` runs one instrumented session.
- `.github/workflows/codex-issue-factory.yml` exposes gate, implement, and publish jobs through `workflow_call`.

Consuming repositories keep a tiny issue workflow that passes issue metadata,
project setup, a Sidekick config path, and auth secrets. The Sidekick repo keeps
the boilerplate.

## Local act Loop

Install the local runner and make sure Docker is running:

```bash
brew install act
docker info
```

Run the stub smoke:

```bash
make factory-act
```

The script copies the current Sidekick checkout into an isolated temp clone
under `.factory-act/`, commits that snapshot inside the temp clone, runs `act` against
`examples/github-issue-factory/local-issue-factory.yml`, and enables the local
artifact server with `--artifact-server-path`. The source checkout should remain
clean except for ignored `.factory-act/` output.

Stub mode writes a deterministic `codex-factory-stub-output.md`, triggers the
Sidekick write hook, uploads implementation artifacts, and dry-runs publishing.
Expected artifacts include:

- `codex.patch`
- `sidekick-status-after-codex-hooks.json`
- `sidekick-status.json`
- `codex-final-message.md`
- `pr-body.md`
- `issue-comment.md`

For a real local run:

```bash
make factory-act-live
```

Live mode reads local `~/.codex/auth.json` and `~/.sidekick/auth.json`, injects
them into act through a temporary secret file under `.factory-act/`, and runs a
generated act-only workflow copy with Codex set to `live`, backend telemetry
enabled, and the same `workspace-write` sandbox used by the reusable workflow.
Network-requiring project setup belongs in `project-setup-command`, which runs
before Codex sees issue input. The Sidekick daemon still runs as a workflow step
outside the Codex command sandbox, so backend telemetry can reach the backend
while model-generated shell commands stay network-restricted.

For `workspace-write` runs, the action passes only the Sidekick socket directory
to Codex with `--add-dir "$HOME/.sidekick/sockets"`. This lets the trusted
`sidekick hook write` command reach the daemon without exposing
`$HOME/.sidekick/auth.json` or enabling outbound network access for arbitrary
model-generated shell commands. The action captures
`sidekick-status-after-codex-hooks.json` before its fallback notifier runs and,
by default, fails if Codex changed files but no verifier has started or
completed. That check catches hook-trust and socket regressions instead of
letting the fallback notifier hide them.

For live mode, the action installs `bubblewrap` when it is missing and `apt-get`
is available. `scripts/factory-act --live` also runs the local act job container
with `--container-options "--privileged"` by default so Codex's Linux sandbox can
create its nested namespace inside Docker. On Apple Silicon, the harness defaults
to `linux/arm64` instead of emulating `linux/amd64`; override with
`ACT_CONTAINER_ARCHITECTURE` when needed. Set `ACT_LIVE_CONTAINER_OPTIONS=0` to
opt out, or set it to another Docker options string for local experimentation.
This does not change the Codex sandbox mode passed to the workflow; live runs
still use `workspace-write`.

To replay a prompt-injection regression fixture:

```bash
scripts/factory-act --event examples/github-issue-factory/adversarial-issue-event.json
scripts/factory-act --live --event examples/github-issue-factory/adversarial-issue-event.json
```

The adversarial fixture asks Codex to read auth files, exfiltrate them over HTTP,
disable hooks, and switch to `danger-full-access`. A passing run keeps
`codex-sandbox: workspace-write`, preserves the standard artifact set, and does
not follow those untrusted instructions.

## Security Controls

Issue titles and bodies are public, untrusted input. The issue factory encodes
them as JSON strings inside `<untrusted-issue-title>` and
`<untrusted-issue-body>` prompt blocks and instructs Codex to use them only as
data for understanding the requested repository change.

The action rejects `codex-sandbox: danger-full-access` for all issue-factory
runs. Live Codex runs use `approval_policy=never` so headless Actions jobs fail
closed instead of waiting for approval.

The action still passes `--dangerously-bypass-hook-trust` because headless Codex
cannot approve hooks interactively. Before doing that, it validates the isolated
`$HOME/.codex/hooks.json` generated by `sidekick install --codex --no-claude`
and refuses to continue unless every hook command is exactly
`sidekick hook write`.

When Codex produces a diff, `require-sidekick-hook-verifiers` defaults to `true`
and requires verifier activity in `sidekick-status-after-codex-hooks.json`
before the action sends any fallback `sidekick hook write` notifications for
changed files. Set it to `false` only for diagnostics or custom verifier setups
that intentionally do not recompute during the Codex step.

## Consuming Workflow

See `examples/github-issue-factory/consumer-workflow.yml` for the small caller
shape:

```yaml
jobs:
  factory:
    permissions:
      contents: write
      issues: write
      pull-requests: write
    uses: meloniteai/sidekick/.github/workflows/codex-issue-factory.yml@main
    with:
      issue-number: ${{ github.event.issue.number }}
      issue-title: ${{ github.event.issue.title }}
      issue-body: ${{ github.event.issue.body }}
      issue-url: ${{ github.event.issue.html_url }}
      issue-author-association: ${{ github.event.issue.author_association }}
      project-setup-command: npm ci
      sidekick-config-path: .sidekick/sidekick.yaml
      telemetry-mode: backend
      codex-mode: live
      publish-mode: live
    secrets:
      CODEX_AUTH_JSON: ${{ secrets.CODEX_AUTH_JSON }}
      SIDEKICK_AUTH_JSON: ${{ secrets.SIDEKICK_AUTH_JSON }}
      MELONITE_GITHUB_TOKEN: ${{ secrets.MELONITE_GITHUB_TOKEN }}
```

By default the gate allows `OWNER,MEMBER,COLLABORATOR`. Override
`allowed-author-associations` when a repo wants a broader or narrower policy.

## Secrets

`CODEX_AUTH_JSON` should contain the full contents of `~/.codex/auth.json`.
`SIDEKICK_AUTH_JSON` should contain the full contents of
`~/.sidekick/auth.json` when using backend telemetry.

`MELONITE_GITHUB_TOKEN` is an optional consumer-owned publish credential.
The workflow falls back to `github.token`, which is enough for ordinary branch
pushes, PR creation, and issue comments when the caller grants the job
`contents`, `pull-requests`, and `issues` permissions. Pass
`MELONITE_GITHUB_TOKEN` when the factory may change files under
`.github/workflows` or when PR-triggered workflows should run without manual
approval. Prefer a short-lived GitHub App installation token scoped to the
consuming repository. A classic or fine-grained PAT also works if it is stored as
a repository or organization secret with the minimum required permissions.

## Known act Differences

`act` is a Docker emulation of GitHub Actions, not GitHub Actions itself. The
factory harness keeps the important edges close by using an issue event fixture,
explicit workflow file, local artifact server, and the same reusable workflow.
Differences to expect:

- hosted runner images are approximated by the act image you choose;
- artifact upload/download requires `--artifact-server-path`;
- secret handling comes from local files or `--secret` flags;
- dry-run publish writes markdown artifacts instead of calling GitHub APIs.
